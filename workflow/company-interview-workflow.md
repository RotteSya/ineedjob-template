# Company Interview Workflow

Last updated: 2026-06-16

## Purpose

このワークフローは、企業別の調査から面接原稿作成までを毎回同じ品質で進めるための標準手順です。新しい対話で企業名や求人票が出たら、この文書と `profile/candidate-profile.md` を最初に読む。

本ワークフローは以下の3つの常設文書とセットで動く。未読のまま面接稿を書いてはいけない:

- `knowledge/japanese-interview-playbook.md` — 日本の新卒面接の採点機構・段階別戦略・当日プロトコル・逆質問戦略・最新トレンド（前提知識）
- `standards/japanese-delivery-standard.md` — 面接稿の話し言葉基準・文字層（長さ・構造・敬語・自然度Lint・キーワード設計）
- `knowledge/native-delivery-pronunciation.md` — 声音層（アクセント・中国語母語干渉・イントネーション・発音リスクリスト手順）
- `standards/interview-qa-gate.md` — 出荷前の強制検査（全ゲート通過まで「完成」と言わない。重要面接は独立検証サブエージェント）

## Standard Flow

### 1. Input Intake

ユーザーから受け取るもの:

- 企業名
- 求人票URLまたは募集要項
- 応募職種
- 面接段階
- 面接形式
- 作りたい成果物

不足していても止まらず、企業名と職種が分かる場合は調査から始める。

面接段階が分かったら、最初に基本ファイルの「0.5 選考ステージ作戦表」を埋める: 面接官タイプ（人事/現場/役員）の推定、この段階で見られるもの、重点準備3問、逆質問の視座。段階不明の場合は一次想定で作り、二次・最終の差分は `knowledge/japanese-interview-playbook.md` §2 で確認する。

### 1.5 Axis and Motivation Resolution

志望動機の解像度を上げるため、企業研究と面接稿作成の前後で「就活軸」「志望動機」「企業決定軸」を分けて整理する。

> 詳細な方法论は `workflow/axis-motivation-workflow.md`（AI内部ガイド）を参照。この §1.5 は簡易入口。目标公司を指定したら、ユーザーに空欄テンプレを渡さず Claude が正本＋調査を読んで考える。**別ファイルは作らない**。結果は本ワークフロー Step 12 の2ファイルに溶け込ませる: 軸テーブル・志望動機4層・企業決定軸/比较スコア・Fit Map は基本ファイル（調査レポート）へ、志望動機・就活の軸・ガクチカ・自己PR の話せる回答は統合版（面接稿）へ。

#### 1.5.1 前提: 企業探しの軸と意思決定の軸を混ぜない

- **企業探しの軸**: 応募企業を効率よく選ぶための判断軸。志望動機の土台になる。
- **意思決定の軸**: 複数社で入社先を迷う時に、最終的に比較するための判断軸。
- 両者は重なる部分もあるが、完全には同じではない。面接官やOB/OGの印象、口コミ、なんとなくの好感だけで判断せず、公式情報、求人票、面接で得た事実、選考体験談を分けて確認する。
- 情報不足で点数が付けにくい項目は、逆質問や追加調査のTODOへ回す。

#### 1.5.2 就活軸の整理

企業別原稿を書く前に、候補者側の「企業探し軸」を3つまで言語化する。**正本フォーマットはユーザー提供のFMT「軸・志望動機の整理FMT〜解像度が上がるフォーマット〜」**。各軸は以下4要素で整理する（列を勝手に変えない）:

| No. | 就活軸 (企業探し軸) | 軸の詳細: どんな状態なら満たされるか | なぜこの軸を求めるか（満たされると自分にとって何が良いか） | 優先順位 |
| --- | --- | --- | --- | --- |
| 1 |  |  |  |  |
| 2 |  |  |  |  |
| 3 |  |  |  |  |

> **漂移防止（重要・過去のミスの教訓）**: この表を「就活軸 / この会社で満たされる理由 / 接続経験 / 優先順位」のような**会社適配表に作り替えてはいけない**。FMTは「候補者本位の軸（なぜ自分が求めるか）」と「企業との接点」を意図的に分けている。第3列は必ず**候補者本位の理由**（満たされると自分にとって何が良いか）。会社でどう満たされるか・接続経験・固有事実は §1.5.3 の4層（②業界・③職種・④会社）と Step 7 Fit Map で扱う。

書く時の注意:

- 「成長できる環境」「グローバルに働ける」だけで止めず、具体的にどんな業務、チーム、顧客、評価指標があれば満たされるかまで書く。
- `profile/career-criteria.md` の母軸（各軸に「なぜ求めるか」を明記済み）から選び、`profile/candidate-profile.md` の Target roles と Core Positioning に接続する。
- 面接で深掘りされる前提で、「なぜその軸が自分にとって大事か」まで答えられる状態にする（FMTの狙い③）。就活軸は会社ごとに作り替えず、優先順位・言い換え・主証拠だけ変える。

#### 1.5.3 志望動機の4層ブラッシュアップ

志望動機は、以下の順番で考える。途中の層が弱いと「なぜこの会社か」が薄くなるため、面接稿の前に必ず確認する。

| Layer | 問い | 確認すること |
| --- | --- | --- |
| 1. 社会に出てやりたいこと | 何を実現したいか | 具体的に誰のどんな課題をどう変えたいか。明確でない場合は Layer 2 以降から仮説を作る。 |
| 2. なぜこの業界か | なぜ他業界ではなく、この業界なら実現できるのか | 業界構造、顧客接点、成長領域、候補者の経験との接点。 |
| 3. なぜこの職種か | なぜ他職種ではなく、この職種なら実現できるのか | 業務内容、期待成果、使える強み、入社後に担う役割。 |
| 4. なぜこの会社か | なぜ他社ではなく、この会社で実現できるのか | 事業、顧客、強み、文化、配属可能性、具体的な仕事内容、誰にどんな影響を与えられるか。 |

企業別の「なぜこの会社か」は、Layer 4 だけを単独で書かない。Layer 1-3 とつながるようにする。

#### 1.5.4 企業決定軸と比較スコア

最終面接、内定承諾、複数社比較の段階では、企業決定軸を3つまで整理し、10点満点で比較する。

| 企業決定軸 | 軸の詳細 | 優先順位 | 対象企業 点数 | 点数理由/確認済み情報 | 比較企業 点数 | 点数理由/確認済み情報 | 情報不足/逆質問TODO |
| --- | --- | --- | --- | --- | --- | --- | --- |
|  |  |  |  |  |  |  |  |

使い方:

- 点数そのものより、「なぜその点数にしたか」を重視する。
- 企業ごとの情報量の偏りに気づくため、点数理由には情報源を簡潔に書く。
- 甲乙つけがたい場合は、優先順位の高い軸を重く見る。
- 企業決定軸は統合版面接稿には入れない。ただし、「他社の選考状況」「第一志望ですか」「入社意思」に関する回答の裏側ロジックとして使う。

### 2. Company Research

調査する項目:

- 事業内容
- 主力サービス/プロダクト
- 収益モデル
- 顧客セグメント
- 経営理念、ミッション、バリュー
- 中期経営計画、成長戦略、重点投資領域
- 競合企業、業界内ポジション、差別化要素
- 海外展開、海外市場・本人の強み領域との接点
- AI/DX/デジタルマーケティングとの接点
- 採用ページの人物像
- 社風、働き方、評価制度、口コミで見える強み/懸念
- 求人票の必須要件、歓迎要件、業務内容
- 直近のニュース、IR、プレスリリース
- 就活会議など就活生体験談ソースから得られる過去の面接質問、選考フロー、内定者傾向

調査ソースの優先順位:

1. 公式一次ソース: 企業公式サイト、採用ページ、IR、プレスリリース、中期経営計画、有価証券報告書
2. 就活生体験談 (必ず確認する。日本新卒就活で最も具体的な質問データが取れるソース群):
   - 就活会議 (shukatsu-kaigi.com): 過去の面接質問、選考フロー、ES通過例、内定者の経歴傾向の主要ソース
   - ONE CAREER: 選考体験記、ES例、業界別ガイド
   - みん就 (みんなの就職活動日記): 掲示板形式で最新の選考動向
   - Unistyle、外資就活ドットコム: ES実例、業界研究 (該当業界の場合)
3. 社員口コミ: OpenWork、ライトハウス、Glassdoor
4. 業界ニュース、ビジネスメディア: 日経、東洋経済、ITmedia 等

就活体験談から必ず抽出する項目:

- 過去2-3年で実際に聞かれた質問リスト (一次/二次/最終で分類)
- 選考フロー全体像 (ES → 適性検査/SPI → 各面接段階)
- 面接段階別の質問傾向の変化
- 内定者の志望理由パターン、評価された自己PR軸
- 落ちた人の典型回答パターン
- 圧迫質問・想定外質問のパターン
- 逆質問の頻出テーマ

就活会議系ソース利用時の注意:

- 投稿者の主観バイアスがあるため、必ず複数ソースでクロスチェックする
- 3年以上前の投稿は選考方針が変わっている可能性が高い。直近の投稿を優先
- 新卒/第二新卒/中途/インターンの区別を必ず確認
- 「内定者」「最終面接落ち」「一次面接落ち」のラベルで重み付けする
- 個別の質問を丸暗記するのではなく、質問の意図と評価軸を抽出する
- 抽出した質問は Step 3 で分析評価し、Step 5 のRisk Checkと Step 9 の質問リストに反映する

会社情報は変わりやすいため、企業研究では原則として最新情報を確認する。

### 3. Interview Question Analysis and Custom Answer Drafting

Step 2 で就活会議等から抽出した実問と網友回答を、評価して自分用のカスタム回答へ落とし込む合成工程。

#### 3.1 信頼度ランク付け

各実問と網友回答に以下のランクを付け、優先順位付けの基礎にする:

- **A**: 直近2年 + 内定者 + STAR形式の詳細回答 → 最優先で参考
- **B**: 直近2年 + (内定者 or 詳細回答) のどちらか → 参考
- **C**: 3-5年前 or 投稿者属性不明 → 補助情報
- **D**: 5年以上前 / 1行コメント / 不明属性 → 参考程度のみ

抽出フォーマット:

| 質問 | 出典 | 投稿年 | 面接段階 | 投稿者属性 | 網友回答要約 | 信頼度 |
| --- | --- | --- | --- | --- | --- | --- |

#### 3.2 網友回答の3観点評価

各代表回答について以下を必ず書く:

- **優点**: STAR構造、数字根拠、企業理念との整合、独自性のうちどれが効いているか
- **不足**: テンプレ感、深掘り耐性の弱さ、数字不在、求める人物像との乖離
- **テンプレ被りリスク**: 同じ角度の回答が複数投稿で見られる場合は「高」と記録

#### 3.3 既知リスクとの接続

抽出した質問を `profile/candidate-profile.md` の Interview Risk Defense と照合:

- 既存のリスク防御（`profile/candidate-profile.md` の Interview Risk Defense に定義したもの。例: キャリアブランク、新卒+職歴、過去の失敗、数字、ビザ、専門深掘り）で対応可能か
- 企業特有の新リスクが発見された場合は Step 5 の Risk Defense テーブルに追記する

#### 3.4 求める人物像との自己チェック

カスタム回答を書く前に、以下を必ず通す:

- Step 2 で抽出した企業の「求める人物像」キーワードと、自分の回答キーワードが対応しているか
- 日本職場の隠性要求 (誠実性、謙虚さ、協調性、論理性、長期視点、報連相、PDCA) のうち、その企業が特に重視しそうな要素を含んでいるか
- 本人の差別化トークン（`profile/candidate-profile.md` の Core Positioning）を1つ以上組み込んでいるか
- `profile/candidate-profile.md` の Packaging Warnings に違反していないか（具体的な禁止項目は profile を正本とする）

#### 3.5 面接段階別の分類

抽出した質問を 一次 / 二次 / 最終 に分類し、各段階で重点準備すべき3問を決める:

- 一次: 学チカ、自己PR、志望動機の基礎チェック中心
- 二次: 業務理解、適性、コミュニケーション能力のマッチング判定
- 最終: 入社意欲、人物像、長期視点の覚悟確認

#### 3.6 カスタム回答テンプレート (高優先3-5問)

最高頻 or 最高難度の3-5問について、以下のフルテンプレートを埋める:

```markdown
### Q[番号]. [質問内容]

**実問の出典と信頼度**:
- 出典: [サイト名] ([投稿年月])
- 投稿者属性: [内定者 / 最終落ち / 一次落ち / 不明]
- 信頼度: [A/B/C/D]
- 面接段階: [一次 / 二次 / 最終]

**網友の典型回答パターン**:
- パターンA: [構造の要約]
- パターンB: [構造の要約]

**典型回答の評価**:
- 優点: [どこが効いているか]
- 不足: [深掘り耐性の弱さ / テンプレ感 / 数字不在 等]
- テンプレ被りリスク: [高 / 中 / 低]

**カスタム高得点回答 (話し言葉化前の骨子)**:

主張:
理由:
具体エピソード (STAR):
学び:
当社で活かす接続:

**深掘り想定2層**:
- Q.1 [想定される追撃質問] → 防御角度
- Q.2 [更なる深掘り] → 防御角度

**差別化チェック**:
- 本人の差別化トークン（candidate-profile の Core Positioning）のどれを組み込んだか
- 求める人物像のどのキーワードに対応しているか
```

#### 3.7 ドラフト原則

- 網友回答は **構造 (型)** のみ参考にする
- 内容は必ず自分の経歴（`profile/candidate-profile.md` の Story Bank / 職務経歴）で置換する
- 差別化要素を1つ以上組み込む
- 深掘り想定2層分まで防御線を準備する

#### 3.8 避けるべき運用

- 網友回答を丸ごと暗記して再現する (面接官に既視感を持たれる)
- 古い投稿 (3年以上前) を最新情報として扱う
- 1件だけのサンプルで「この会社の質問」と決めつける
- 投稿者属性を確認せずに採用する
- 自分の経歴と無理に接続して矛盾を生む
- 就活会議で見つからなかった質問は対策不要と判断する (基礎質問は必ず来る)

### 4. Requirement Extraction

求人票から以下を分ける:

- Must: 必須要件
- Want: 歓迎要件
- Hidden Need: 求人票に直接書かれていないが、事業内容から推測できる期待
- Risk: 候補者側で弱く見える点
- Hook: 面接官に刺さる候補者側の証拠

### 5. Risk and Defense Check

面接稿を書く前に、必ず `profile/candidate-profile.md` の Interview Risk Defense を確認する。さらに Step 2 で就活会議等から収集した「実際に聞かれた質問リスト」と Step 3 の分析結果を突き合わせ、本人の弱点に当たる質問が頻出している場合は優先度を上げる。企業別に以下を埋める:

| Risk | Likely Question | Defense Angle | Evidence Needed |
| --- | --- | --- | --- |
| キャリアの空白/ブランク（あれば） | この期間は何をしていましたか | 本人の Interview Risk Defense の方針 | 学習/準備/転換に関する事実 |
| 新卒枠と既存職歴のギャップ（あれば） | なぜ新卒採用ですか | 学びながら早期貢献する姿勢 | 在学状況、入社時期 |
| 過去の失敗・頓挫した取り組み（あれば） | 失敗から何を学びましたか | 学びと次の行動への接続 | 該当プロジェクトの事実 |
| 数字実績 | 測定方法は何ですか | 期間、母数、測定指標、自分の担当範囲を明確化 | 管理画面/資料/記憶メモ |
| 在留資格・ビザ（留学生の場合） | 入社時に問題ありませんか | 留学ビザから就労可能な在留資格へ変更予定 | 大学・入管手続き確認 |
| 専門/技術の深掘り | （専門外・限界を問う質問） | 専門範囲を正直に＋学習姿勢 | 説明可能範囲・証拠 |

Step 3 で発見された企業特有の新リスクは、この表に追記する。

### 6. Evidence Checklist

数字や受賞を使う場合、`profile/evidence-ledger.md` を確認し、面接前に以下を準備する:

- 実績数字ごとの期間
- 対象範囲: 個人、チーム、プロジェクト全体、アカウント群
- 測定方法: 管理画面、月次レポート、投稿データ、応募フォーム等
- 自分の担当範囲
- チーム成果として話すべきか、個人貢献として話すべきか
- 証拠資料がない場合の控えめな言い換え
- 面接前72時間以内に、企業別に必要な実物資料の発掘TODOを作る（候補は `profile/evidence-ledger.md` の各実績に紐づく証拠: スクリーンショット、投稿/成果物URL、当時のレポート、提案書、受賞/表彰資料、各種記録など）。

企業別の基本ファイルには、以下の形で `profile/evidence-ledger.md` から使う数字だけを転記する:

| Claim Used | Evidence Status | Measurement Defense | Personal Contribution | Conservative Phrasing | Next Action |
| --- | --- | --- | --- | --- | --- |
|  |  |  |  |  |  |

### 7. Fit Map

企業要件、候補者経験、Step 1.5 の就活軸を接続する。形式:

| Company Need | Candidate Evidence | Story | Japanese Angle |
| --- | --- | --- | --- |
| （企業の要件1） | （profile/candidate-profile.md の該当経験） | （Story Bank の一件） | （日本人面接官に伝わる見せ方） |
| （企業の要件2） |  |  |  |

> 考え方:「企業が求めること」→「それを満たす本人の経験」→「具体的な一件（Story Bank）」→「日本語での見せ方」。必ず `profile/candidate-profile.md` の事実に基づく。盛りすぎない。

Fit Map の後に、志望動機の4層ロジックを企業別に埋める:

| Layer | この企業向けの答え | 根拠/接続する経験 | 弱い場合の追加調査 |
| --- | --- | --- | --- |
| 社会に出てやりたいこと |  |  |  |
| なぜこの業界か |  |  |  |
| なぜこの職種か |  |  |  |
| なぜこの会社か |  |  |  |

この表で矛盾や空白がある場合、面接稿の「志望動機」は先に書かず、Step 2 の追加調査または短い discovery で埋める。

### 8. Story Selection

`profile/candidate-profile.md` の Story Bank / Work Experience / Major Projects から、この企業の求める人物像に合うSTAR素材を選ぶ。

- 各 Story に「どの母軸・どの企業要件に効くか」を一言添える。
- 数字を使う Story は `profile/evidence-ledger.md` の Status と控えめ表現を確認する。
- 企業別に3本程度へ絞る（多すぎると一つひとつが浅くなる）。

If `profile/candidate-profile.md` still has unresolved Discovery TODOs for collaboration/Hou-Ren-Sou/feedback improvement, run one short multiple-choice discovery session before drafting final scripts.

### 9. Draft Interview Script

作るべき基本セット:

- 1分自己紹介
- 志望動機
- 就活の軸（企業選びの軸）は何ですか
- 自己PR
- 学生時代に力を入れたこと
- 前職で頑張ったこと
- 失敗経験
- チームで困難を乗り越えた経験
- 長所と短所
- なぜ日本で働きたいか
- 本人の経歴特有の「なぜ」質問（進学・転職・地域選択などの理由。`profile/candidate-profile.md` の背景から生成する）
- なぜ新卒採用で応募するのか
- キャリアの空白期間について（あれば）
- なぜこの会社か
- 5年後/10年後のキャリアビジョン
- 他社の選考状況
- 当社の課題は何だと思うか
- 在留資格・就労ビザについて
- 技術プロジェクトで自分が担当した範囲
- 逆質問

加えて、Step 3 で抽出・分析した **企業特有の高優先3-5問** を上乗せする。基本セットの汎用回答よりも、企業固有質問への回答の方が評価に直結する。

志望動機を書く時は、Step 1.5 と Step 7 の4層ロジックを使い、「社会に出てやりたいこと → 業界 → 職種 → 会社」の順に自然につなげる。会社名だけを差し替えられる回答になっている場合は、Layer 4 の事業・職種・顧客・配属可能性の根拠を追加する。

ドラフト時点から以下を守る（後工程での手直しを減らす）:

- 長さは `standards/japanese-delivery-standard.md` §1 の表に従う。デフォルトは30〜45秒。1分級は主要4答（志望動機・自己PR・ガクチカ・前職）だけ。
- 結論まで10秒。フック設計（全部一気に話さない。深掘りされたい所を残す）。
- 自己紹介は30秒版と1分版の2本を作る。
- 主要回答には深掘りサブQ&A（追撃質問＋15〜30秒の返し）を2〜3個付ける。元ネタは基本ファイルの深掘りツリー（コンピテンシーチェーン: 役割→なぜ→他の選択肢→周囲→測定→今ならどうする）。
- 「今ならどうする?」への返しは全主要エピソードに必ず用意する（最頻出の隠れ追撃）。

### 10. Reverse Question Strategy

逆質問は評価項目として扱う。必ず3階層で準備する:

1. Business理解: 事業、競合、中期経営計画、海外展開に関する質問
2. Role理解: 入社後の業務、期待成果、チーム体制、評価指標に関する質問
3. Growth/定着: 育成、キャリアパス、長期的な貢献に関する質問

サンプル:

- 御社が今後注力されている領域の中で、新卒社員にも早期から関われる可能性があるテーマはありますか。
- 海外展開や注力されている領域に関して、現場で特に難しいと感じている課題はどのような点でしょうか。
- 入社後半年から1年で、どのような状態になっていれば「順調に成長している」と評価されますか。
- この職種で成果を出している方に共通する行動や考え方はありますか。
- AIやデジタルツールの活用について、現場で今後改善したい業務はありますか。
- 配属後、他部署や海外拠点と連携する機会はどの程度ありますか。

避ける質問:

- 調べれば分かる基本情報だけを聞く
- 給与・残業・福利厚生だけに寄せる
- 「勉強できますか」だけで終わり、貢献意欲が見えない質問

Step 2-3 で抽出した「網友が高評価された逆質問テーマ」があれば、それを参考にして自分用にアレンジする。

Step 1.5 の企業決定軸スコアで情報不足になった項目は、逆質問候補へ変換する。例: 点数理由が口コミだけの場合は、現場の評価指標、配属後の業務範囲、育成体制、海外/AI/DX領域への関与可能性を確認する質問にする。

### 11. Spoken-Japanese Pass（話し言葉化）

`standards/japanese-delivery-standard.md` §7 の Humanize手順を統合版に対して実行する（外部 `/humanizer` コマンドがある環境では併用してよいが、合否判定は同基準が正本）:

1. 構造確定（結論まで10秒・ワンメッセージ・フック設計）
2. 書き言葉→話し言葉変換（§4）と長さ実測（§1）
3. 敬語NG表（§3）と自然度Lint 12項目（§5）の機械的検査
4. 音読シミュレーション（舌が回らない語・息継ぎ不能文の排除）
5. 各回答に「▼キーワード: 3〜4語」行を付与（§6。全文暗記禁止の仕組み）

方向性は従来どおり: 謙虚だが弱く見えない / 「私はすごい」ではなく「経験から学び、次に活かす」/ `profile/candidate-profile.md` の Packaging Warnings 通り盛らない / 母語（中国語など）の直訳調を残さない。

#### 11.1 声音層パス（発音・アクセント）

文字層（§11）を終えたら、`knowledge/native-delivery-pronunciation.md` に従って声音層を準備する（母語干渉＋留学生という、留学生面接の最大の伸びしろ）:

1. 面接稿の主要回答と固有名詞から、崩れやすい語（社名・製品名・術語・本人氏名・母語由来語）を抽出。
2. §6手順で OJAD スズキクン/単語検索/辞典で読みを確認（個別語の核は記憶で断定しない）。実査できなければ「要確認」と明記。
3. §7様式で基本ファイルの**発音リスクリスト**を作る。読みにくい語は言い換え案を併記。
4. 句末上がり・長音短縮・四声化の癖を納品時の練習指示に反映。

### 11.5 QA Gate（出荷前検査）

`standards/interview-qa-gate.md` の Gate A〜D（+C9 声音層）を全て通す。1項目でもFailなら修正して再検査。検査結果と実測メモ（最長回答字数・文末分布など）は基本ファイル末尾の「QAゲート記録」に残す。**全ゲートPassまで、ユーザーに「完成」と報告しない。**

**大手企業・二次以降・最終面接の稿は、書いたセッションとは別の独立検証サブエージェントに Gate A〜D を実測させる**（手順は qa-gate.md「独立検証サブエージェントの回し方」）。検証者には追認でなく Fail探しを依頼する。自己検査で済ませる場合（軽い一次のみ等）はその理由を記録に1行残す。

納品時には毎回、練習指示を添える: 音読3回（3回目はキーワード行のみ）→ 録音1回 → 主要4答のキーワード再現テスト。

### 12. Output File Naming and Role Separation

二種類のファイルを明確に役割分担する。混ぜないこと。

新しい会社の面接準備では、原則として以下のテンプレートから開始する:

- 基本ファイル: `docs/_templates/base-interview-template.md`
- 統合版ファイル: `docs/_templates/integrated-interview-template.md`
- フィードバックファイル: `workflow/feedback-template.md`

#### 12.1 基本ファイル `docs/<company>/<company>_interview_<YYYY-MM-DD>.md`

リサーチと分析の作業場。以下のメタ情報を全て含む:

- 選考ステージ作戦表 (Step 1: 面接官タイプ・段階別の見られ方・重点3問)
- 会社調査 (Step 2)
- 就活軸、志望動機4層ロジック、企業決定軸/比較スコア (Step 1.5, Step 7)
- Q&A Archive (Step 3): 出典、信頼度ランク (A/B/C/D)、投稿年、投稿者属性、網友回答の3観点評価 (優点/不足/テンプレ被りリスク)、求める人物像マッピング、既知リスクとの接続、面接段階別分類、深掘り想定2層、差別化チェック
- 要件抽出 (Step 4)
- リスク防御テーブル (Step 5)
- Evidence Checklist (Step 6): `profile/evidence-ledger.md` から使う数字だけを転記
- Fit Map (Step 7)
- Story Selection (Step 8)
- 深掘りツリー (Step 9: コンピテンシーチェーン6追撃×主要エピソード)
- 発音リスクリスト (社名・術語の読みと言い換え)
- 参照URL一覧
- QAゲート記録 (Step 11.5 の検査結果と実測メモ)

このファイルはユーザーが面接当日に見るためのものではない。準備過程の根拠と監査ログ。

#### 12.2 統合版ファイル `docs/<company>/<company>_interview_integrated_<YYYY-MM-DD>.md`

面接当日にユーザーが直接見て使う版。純Q+A形式のみ。以下は **入れない**:

- 出典、信頼度ランク、投稿者属性、投稿年
- 網友回答パターン、テンプレ被りリスク評価
- 求める人物像マッピング、既知リスク接続表
- 深掘り想定の解説文、差別化チェック
- 企業決定軸の点数表、比較スコア、情報不足TODO
- 戦略メモ、初回面接の戦い方
- 日本語を自然にするコツ、話し方トレーニング
- 詰まった時の共通リカバリー句 (それは `knowledge/japanese-interview-playbook.md` §6 にある)
- 参照URL一覧

含めるのは以下の3要素だけ。すべて「当日口に出す・当日目に入れる」ためのもの:

1. **質問の見出し + 話せる長さの回答** (`standards/japanese-delivery-standard.md` 準拠。複数バージョン可: 自己紹介は30秒版+1分版)
2. **▼キーワード行** (回答末尾に3〜4語。当日直前に見るのはこの行。全文暗記の防止装置)
3. **深掘りサブQ&A** (主要回答の直下に、追撃質問+15〜30秒の返し。これも純Q&Aの一部であり解説ではない)

雛形は `docs/_templates/integrated-interview-template.md` (構成例はテンプレート自体を参照)。

#### 12.3 フィードバックファイル `docs/<company>/<company>_feedback_<YYYY-MM-DD>.md`

面接後の実問・回答・改善ログ。`workflow/feedback-template.md` を雛形にする。

#### 12.4 ファイル運用ルール

- 基本ファイルには Step 1.5 の軸/志望動機整理と、Step 2-3 で抽出・分析した Q&A Archive セクションを必ず含める。
- 統合版は基本ファイルから「面接で口に出す部分だけ」を抽出して作る。話し言葉化（Step 11）とQA Gate（Step 11.5）は統合版に対して適用する。
- 新しい会社の資料は必ず `docs/<company>/` フォルダを作って保存する。
- 既存企業の資料がある場合は、重複作成よりも既存ファイルへの追記/更新を優先する。
- 単独の `_research_` ファイルは、調査量が多い場合だけ作る。

### 13. Post-Interview Feedback Loop

面接後は、必ず `workflow/feedback-template.md` を使って以下を記録する:

- 実際に聞かれた質問
- 自分が答えた内容
- 面接官の反応
- 詰まった質問
- 次回までに直す回答
- 数字や証拠で追加準備が必要な点
- 逆質問の反応

このログを次の会社の面接稿に反映し、求職活動自体でPDCAを回す。さらに、Step 3 の Q&A Archive で予測した質問と実問の差分を確認し、就活会議系ソースの精度評価にも使う。数字や受賞について追撃質問が出た場合は、`profile/evidence-ledger.md` の Status / Next Action も更新する。

内容だけでなく**話し方**も記録する: 詰まった箇所、長すぎた回答、棒読みになった回答、聞き返された語（→基本ファイルの発音リスクリストへ追記）。繰り返し出る話し方の癖は `standards/japanese-delivery-standard.md` の Lint 項目として昇格させる。
