# Japanese Delivery Standard（面接話法・原稿基準）

Last updated: 2026-06-13

## これは何か

面接稿（`<company>_interview_integrated_<date>.md`）の日本語を「面接官が聞いて自然で、評価される話し言葉」にするための**強制基準**。すべての面接稿は、出荷前に本基準と `standards/interview-qa-gate.md` を通過しなければならない。

旧フローの `/humanizer` 外部コマンドへの依存はここで解消する。`/humanizer` が使える環境では併用してよいが、**最終的な合否判定は本基準が正本**。

> 設計思想: 面接官は「完璧な作文」ではなく「会話が成立する人」を採点している。リクルート就職白書系の調査では採用基準の第1位は一貫して「人柄」。原稿の役目は、暗記して再生することではなく、**キーワードと構造を体に入れて、当日その人の言葉で話せる状態を作る**こと。

---

## §1 長さ基準（時間×字数）

話速の基準は **1分間=300字**（NHKアナウンサー基準。緊張すると速くなるため、原稿は300字/分で設計し、本番は「少しゆっくりすぎるかな」と感じる速度で話す）。

| 質問タイプ | 目標時間 | 字数目安 | 備考 |
| --- | --- | --- | --- |
| 自己紹介（指定なし） | 30〜40秒 | 150〜200字 | 別途1分版（280〜330字）も常備。「1分で」と指定された時用 |
| 志望動機 | 60〜90秒 | 300〜420字 | 4層ロジックを話し言葉に圧縮 |
| 自己PR / ガクチカ / 前職で頑張ったこと | 約60秒 | 280〜350字 | 「1分半で」指定に備え420字版も可 |
| 通常質問（長所短所・キャリアビジョン・なぜ日本 等） | 30〜45秒 | 150〜220字 | ここが一番作りすぎ事故が多い |
| 深掘り追撃への回答 | 15〜30秒 | 80〜150字 | 会話のラリーを保つ |
| 事実確認系（ビザ・他社状況・勤務地 等） | 15〜30秒 | 80〜150字 | 端的に。長いと不安に見える |
| 逆質問（1問あたり） | 〜15秒 | 〜80字 | 前置きを短く |

**絶対上限**: どの回答も**90秒/450字を超えない**。1分半を超える一方的な発話は「要点をまとめる力がない」と減点される（就活の未来調査: 86.5%の学生が30秒〜1分半に収めている）。

**結論まで10秒ルール**: 第一文＝質問への直接の答え。10秒（約50字）以内に結論が出ない回答は書き直し。

## §2 構造基準

| 型 | 使う質問 | 構造 |
| --- | --- | --- |
| **PREP** | 志望動機、軸、長所短所、考えを問う質問 | 結論→理由→具体例→結び |
| **STAR圧縮** | ガクチカ、前職、失敗、チーム経験 | 結論（何をして何を得たか）→状況1文→行動2文→結果＋学び |

原則:

1. **ワンメッセージ**: 1回答に主張は1つ。強みを2つ以上並べたら分割する。
2. **フック設計（最重要）**: 一気に全部話さない。深掘りされたい所をあえて言い切らずに残す。面接は「演説」ではなく「会話のラリー」。フックの形:
   - 「いくつか工夫したのですが、特に効果が大きかったのは〜です」（→他には?と聞かせる）
   - 「結果として最優秀賞をいただきました」（→何が評価された?と聞かせる）
   - 数字を1つだけ出す（→どう測った?と聞かせる。回答は evidence-ledger で防御済みの数字に限る）
3. **数字は1回答に1〜2個まで**。数字の羅列は録音再生に聞こえる。
4. 基本ファイル側に、各主要回答の**深掘り2層**（追撃質問→15〜30秒の返し）を必ず併設する。面接稿には深掘りQ&Aとして載せてよい（それも純Q&Aの一部）。

## §3 敬語基準（NG表）

完璧な敬語より「です・ます」の徹底＋落ち着いた言い直しが評価される。間違えたら「失礼いたしました」と言い直せばよい。ただし以下は原稿段階でゼロにする:

| × NG | ◯ 正 | 種類 |
| --- | --- | --- |
| おっしゃられる / 拝見させていただきました | おっしゃる / 拝見しました | 二重敬語 |
| 〜になります（履歴書になります） | 〜です | バイト敬語 |
| 〜のほう（資料のほう） | 〜（資料） | バイト敬語 |
| よろしかったでしょうか | よろしいでしょうか | バイト敬語 |
| 了解しました / わかりました | 承知しました / かしこまりました | 対上NG |
| なるほど / 確かに | おっしゃるとおりです | 対上NG |
| 大丈夫です | 問題ありません | 曖昧 |
| 参考になりました | 勉強になりました | 失礼 |
| すみません / ごめんなさい | 申し訳ございません | 軽い |
| 見れる / 食べれる | 見られる / 食べられる | ら抜き |
| 〜させていただいております（許可不要の場面） | 〜しております | させていただく濫用 |
| 貴社（口頭） | 御社（口頭。書面は貴社） | 混同 |
| 僕 / 自分 | 私（わたし／わたくし） | 一人称 |
| お父さん / お母さん | 父 / 母 | 身内呼称 |
| 部長様 / 社長様 | 部長 / 社長の〇〇様 | 役職+様 |

「させていただく」は1回答に最大1回。迷ったら「〜しております」「〜しました」。

## §4 書き言葉→話し言葉 変換表

原稿は最初から話し言葉で書く。ESの文体をそのまま読むと一発で「書いたものを暗記した人」になる。

| 書き言葉 | 話し言葉 |
| --- | --- |
| しかし / だが | ですが / ただ |
| 〜および〜 | 〜や〜 / 〜と〜 |
| 〜を実施しました / 遂行しました | 〜をしました / 〜を進めました |
| 〜することができました | 〜できました |
| 〜という結果を達成した | 〜という結果につながりました / 〜を出せました |
| 第一に / 第二に | まず / 次に / あとは |
| 〜である | 〜です |
| 〜のような形で / 〜という形で | 〜のように / （削る） |
| 〜を把握し | 〜をつかんで / 〜を理解して |
| 非常に | とても / かなり |
| 〜に貢献したいと考えております（連発時） | お役に立ちたい / 力になりたい / 活かしたい と混ぜる |

文の長さ: **1文は50字以内目安（話して10秒以内）**。読点「、」が3つを超えたら2文に割る。入れ子の複文（「〜が、〜ので、〜ですが、〜」）は禁止。

## §5 自然度Lint（翻訳調・暗記調の検出）

面接稿の完成前に、以下12項目を機械的にチェックする。1つでも該当したら修正:

1. **文頭「私は/私の」**が全文の3割超 → 日本語は主語を落とす。「私の強みは〜」で始まる回答は1稿に2つまで。
2. **同一文末の3連続**（〜です。〜です。〜です。／〜ました。×3）→ 文末を散らす: 〜です / 〜ました / 〜ています / 〜と考えています / 〜と感じています / 体言止めは使わない（話し言葉では不自然）。
3. **「〜たいです」が1回答に3回以上** → 願望の連発は幼く聞こえる。「〜つもりです」「〜と考えています」と交ぜる。
4. **「〜と考えています」の濫用** → 1回答に1回まで。
5. **「〜することができます」** → 「〜できます」。
6. **「〜を通じて」「〜という形で」の頻用** → 「〜の中で」「〜から」に散らすか削る。
7. **接続詞パターンが全回答で同型**（全部「まず→そこで→結果として」）→ 面接官は複数回答を続けて聞く。型の使い回しは暗記とバレる。「最初は / その中で / 結局」「きっかけは / ただ / 最終的に」などに散らす。
8. **硬い漢語の連発** → 構造化・属人的・非対称性などの分析語は1回答に1個まで。2個目からは「人によってばらばら」「情報の差」など和語に開く。
9. **全回答が「御社でも〜貢献したいです」で終わる** → 結びを変える: 「〜が御社でも活かせると考えています」「〜に挑戦したいです」「〜を大事にしていきたいです」。
10. **40字を超える名詞句**（「地域の中小店舗向けの来店促進データ分析支援サービス」等）→ 話す時は分解する: 「地域のお店向けに、来店を増やすためのデータ分析を支援するサービスです」。
11. **回答冒頭の「はい、」**: 全回答に付ける/全部省くのは機械的。6割程度に付ける。質問を一部復唱して始めるのも自然（「私の強みは、と言いますと」）。
12. **カタカナ語が1文に3個以上** → 日本語に開くか文を割る。

加えて**中国語話者の翻訳調**として特に注意:

- 「〜の方面」「〜に対して理解がある」→「〜について」「〜をよく知っている」
- 大きな抱負表現（「必ず」「絶対に」「全力で」）の連発 → 1稿1回まで。日本の面接は控えめ+具体が信頼される。
- 中国固有名詞には**日本側アナロジーを一言添える**: Bilibili→「日本でいうニコニコ動画とYouTubeを合わせたような動画サイト」/ WeChat→「中国版LINE」/ 百度贴吧→「日本でいう5ちゃんねるやRedditのような掲示板」/ 吧主→「コミュニティの管理人」。アナロジーは初出時のみ。

## §6 キーワード設計（暗記棒読みの根絶）

**全文暗記は禁止**。暗記調は「熱意が低い」「想定外に弱い」と読まれる。代わりに:

- 各回答に**キーワード3〜4個**（名詞＋数字＋動詞）を設計し、面接稿の回答末尾に1行で付ける:
  `▼キーワード: 10アカウント / データで改善 / 新人2週間 / 仕組み化`
- 練習はキーワード行だけを見て話す。3回連続で要旨を再現できたら合格。
- 本番で言い回しが原稿と変わるのは**正常**。守るのはキーワードと順序だけ。

## §7 Humanize手順（旧 /humanizer の内製化）

面接稿を仕上げる時、必ずこの順で実行する:

1. 構造を確定（§2。結論ファースト・フック・ワンメッセージ）
2. §4変換表で話し言葉化、§1で長さを調整（文字数を実測する）
3. §3敬語NG表・§5 Lint 12項目を機械的に検査
4. **音読シミュレーション**: 各回答を頭の中で発話速度で再生し、息継ぎできない文・舌が回らない語（例:「諸条件」「組織横断的」）を書き直す
5. キーワード行を付与（§6）
6. `standards/interview-qa-gate.md` の全ゲートを通す
7. ユーザーへの納品文に「音読3回＋録音1回＋キーワードのみ再現テスト」の練習指示を添える

## §8 非母語話者のデリバリー補正

- **発音リスクリスト**: 面接稿の末尾ではなく基本ファイル側に、その会社で必ず口にする固有名詞・術語の読みリストを作る（社名・製品名・業界の独自語など）。読みが難しい語は言い換え語も併記。
- **通じなかった時の原則**: 同じ語を繰り返さず、**より簡単な言葉に言い換える**。基礎語で言い直せること自体が理解の証明になる。
- 語尾を伸ばさない（「〜でぇ」「〜ですー」禁止）。最初の音をはっきり。緊張時は意識的に1段ゆっくり。
- フィラー（えーと、あのー）は2〜3秒の**沈黙の方がまし**。沈黙は「考えてから話す人」と好意的に取られる。5秒を超えそうなら「少し考えるお時間をいただけますか」。
- N1でも落ちる原因は語学力ではなく「結論までが長い」こと。§1の結論まで10秒ルールが最大の防御。

## §9 文体サンプル（Before → After）

> 以下は**架空のサンプル人物**（留学生「リン」。越境EC運用→日本の大学院）の経歴での例。見るべきは技術（アナロジー・数字の絞り方・フック・結びの着地）であり、内容は自分の `candidate-profile.md` の事実に置き換える。

### 例1: 自己紹介（Before: 書き言葉・390字・特徴の羅列）

> リンと申します。中国の大学で経営学を学んだ後、上海の越境EC企業で約2年間、SNSを中心としたブランド運用に従事しました。10以上のアカウントを対象に、投稿企画、データ分析、多言語対応を担当し、主要アカウントでは月間で大幅なフォロワー増加も達成しました。その後、日本で長期的にキャリアを築きたいと考え、現在は日本の大学院でマーケティングを専攻しております。（以下略）

問題: ESの文体そのまま／数字と専門語の羅列／フックがなく深掘りどころが見えない／「携わりました」「経験しました」の硬さ。

### 例1: 自己紹介（After: 1分版・約300字・話し言葉）

> リンと申します。中国の出身で、大学卒業後、上海の越境ECの会社で約2年働いていました。担当していたのは「小紅書（レッドノート）」という、日本でいうInstagramと口コミサイトを合わせたようなアプリでの、海外向けの発信です。10以上のブランドのアカウントを任されて、データを見ながら投稿を改善する仕事をしていました。
> その後、日本で長くキャリアを築きたいと考えて、今は日本の大学院でマーケティングを学んでいます。在学中には、地元の商店街の集客を支援する学内プロジェクトで、4人のチームのリーダーを務め、来客を増やす施策をまとめて、学内コンテストで優秀賞をいただきました。
> 海外ユーザーの理解と、データで施策を改善する経験を御社で活かしたいと考えています。本日はどうぞよろしくお願いいたします。
>
> ▼キーワード: 上海2年・越境EC / 10アカウント・データ改善 / 商店街プロジェクト・優秀賞 / 海外ユーザー理解

設計意図: 数字は「10」「4人」「2年」だけに絞り、大きな実績はあえて出さない（深掘りで出すための弾）。アナロジー（小紅書→Instagram＋口コミ）で面接官の認知負荷を下げる。結びの挨拶で着地。

### 例2: 強み（Before の典型問題）

> 私の強みは、課題を観察し、データと文脈の両方から改善策を考え、チームで再現できる形に整理できる点です。（〜350字続く）

問題: 強みが3つ並列（観察×分析×仕組み化）でワンメッセージ違反。「再現できる形に整理」が書き言葉。

### 例2: 強み（After: 45秒・約220字）

> はい。強みは、感覚に頼らずデータで考えて、結果を仕組みに変えられるところです。
> 前の会社では、どの投稿が伸びるかが担当者の経験頼みになっていました。私は伸びた投稿のテーマや投稿時間、反応を一つずつ分解して、誰が見ても同じ基準で判断できる一覧表を作りました。その結果、新しく入った担当者が一人で投稿を任せられるまでの期間が、1か月以上から2週間ほどに縮まりました。
> この「属人的なものを仕組みにする」点は、御社の仕事でも活かせると考えています。
>
> ▼キーワード: データで考える / 伸びた要因の分解 / 一覧表 / 新人2週間

## 運用メモ

- 本基準は面接稿（integrated）に適用する。基本ファイル（調査レポート）は書き言葉のままでよい。
- 例外: 企業から「1分で」「3分で」等の指定が既知の場合、その指定が§1より優先。
- 本基準を変更する時は `standards/interview-qa-gate.md` のチェック項目との整合を保つこと。
